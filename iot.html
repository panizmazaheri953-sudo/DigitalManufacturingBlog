01 — The Big Picture
LOOP

The Industry 4.0
Closed Loop.

This is what separates digital manufacturing from traditional manufacturing. The physical FESTO CP Lab produces real data. That data flows into Siemens Insights Hub. Insights Hub feeds into Plant Simulation. The simulation results go back into Insights Hub. The loop is closed and the digital model improves continuously as the physical system runs.

01
Physical World
FESTO CP Lab
Each station has a Programmable Logic Controller (PLC) generating real process data including start times, end times, positions, and many more sensor readings.
Real hardware
02
Gateway
MindConnect Nano
The MindConnect Nano serves as the cloud gateway. It reads PLC datapoints from each station via IP address and streams the data to Siemens Insights Hub in real time at a 1 second cycle.
1s data cycle
03
Cloud Platform
Insights Hub
Siemens Insights Hub stores and processes the live data. Virtual assets mirror each physical FESTO station. Factory Twin maps the data to the Plant Simulation model through calculated recipes.
Live cloud data
04
Simulation
Plant Simulation
Real processing times from Insights Hub are loaded into Tecnomatix Plant Simulation, updating the model with actual data. The simulation now reflects how the real system is performing, not just planned values.
Loop closed ✓
Why this matters for the Phones-R-Us contract
When 25 Bambu P1S printers are running 24/7, the IoT system does not just monitor. It enables safe unattended operation. Fire detection, security alarms, printer failure alerts, and performance monitoring all feed into the same connected ecosystem. If a printer fails at 3am, the system detects it, logs it, and alerts the engineer. The digital twin means the simulation is continuously updated with real production data, so capacity planning stays accurate as the system runs (Tao et al., 2018).

Insights Hub

Week 9 Workshop
Step 1 — Virtual Representation
Aspect Types and Asset Types
The virtual structure of each FESTO station was created in Insights Hub's Asset Manager. Each station receives an Aspect Type containing variables, an Asset Type serving as a template, and an Asset representing the actual virtual station.
  • An Aspect Type was created with 3 variables per station (start, end, position)
  • An Asset Type was created from the aspect template
  • An Asset was created as the virtual twin of each physical station
Step 2 — Physical Connection
MindConnect Nano
The physical FESTO PLC was connected to Insights Hub through the MindConnect Nano gateway. Each station's IP address was mapped to a data source, with 3 datapoints per station streaming at 1 second intervals.
  • Data sources configured using PLC IP addresses per station
  • Station 1 (Front Magazine): 172.21.1.1
  • Station 2 (Measurement): 172.21.2.1
  • Stations 3 to 6 mapped at their respective IP addresses
  • Data acquisition mode set to READ with a 1s cycle time
Step 3 — Data Mapping
Joining Physical to Virtual
The physical datapoints from MindConnect were mapped to the virtual variables in the Asset. This completes the connection, meaning real PLC data now appears as virtual asset variables in the cloud.
  • Each physical datapoint was linked to its corresponding virtual variable
  • All 6 stations were mapped: FM, Measurement, iPick, Output, Press, Oven
  • Data is now visible as live time series in Insights Hub
  • The foundation for Factory Twin and Digital Twin is built on top of this

Week 6 Workshop
AI Enhanced Workflow
Using AI to generate SimTalk code and optimise the simulation
Rather than writing all the Tecnomatix SimTalk code manually, AI was used to accelerate the simulation build (Kusiak, 2018). The production system was described in detail to the AI, covering station processing times, worker shifts, and parallel stations. The AI generated the SimTalk code for setting up experiments and parameter tables. The outputs were then critically verified and adjusted before being applied, following good engineering practice.
01
Describe the system
A full production model was described to the AI, including station processing times, worker pools, shifts, parallel stations, and bottleneck analyser locations.
02
AI generates SimTalk
The AI produced SimTalk code for the Init method, parameter table structure, and experiment manager configuration, accelerating the build significantly.
03
Verify and apply
The AI output was critically reviewed, tested, and corrected before applying. It was not blindly accepted; engineering judgement was applied to validate every output.
What the Experiment Manager Was Used For
Experiment 1
How Many Workers?
The Experiment Manager was used to test different numbers of workers against throughput output. The simulation was run multiple times with the numWorkers variable set to different values, identifying the minimum worker count that still achieves the production target.
  • The variable numWorkers was tested across a range from 1 to 11
  • The output was drain.StatNumIn, representing total parts completed
  • 5 observations per experiment ensured statistical validity
  • The result identified the optimal worker number from data, confirming that 2 workers per shift is sufficient
Experiment 2
Reducing Printer Count
The Experiment Manager was used to test the effect of reducing the printer fleet from 25 to 23 printers. This experiment helped quantify how much buffer capacity the 25 printer configuration provides and whether a smaller fleet could still meet the 50,000 unit target.
  • The variable number of printers was reduced from 25 to 23
  • The output was drain.StatNumIn, representing total parts completed
  • Multiple observations per experiment ensured statistical validity
  • The result confirmed that 25 printers are needed, as 23 printers fall short of the 50,000 target
Experiment 3
Bottleneck Analysis
Each station was linked to a bottleneck analyser in the simulation. Running experiments across different configurations revealed where throughput was most constrained, confirming Station 5 (Oven, 35 seconds) as the FESTO line bottleneck.
  • A bottleneck analyser was connected to every station
  • Station 5 (Oven) at 35 seconds was confirmed as the primary constraint
  • The data confirmed that the total FESTO cycle remains within the 150 second takt requirement
  • The simulation matched physical FESTO measurements

Factory Twin —
Future Implementation.

Proposed Next Steps
📋 Planned Enhancement
The Digital Twin has not yet been fully implemented. The following section describes the proposed architecture and steps that would close the IoT to simulation loop. This represents the next phase of the ENCLOS3D digital manufacturing strategy .

The Digital Twin is planned as the most advanced element of this project. Factory Twin (a Siemens Insights Hub application) would create a live two way connection between the physical FESTO CP Lab and the Tecnomatix Plant Simulation model, fully closing the loop. The steps below outline how this would be implemented.

01
Generate the Mapping File from Plant Simulation
The MindSphereInterface library would be added to the Plant Simulation model. The interface would be configured to select the FESTO stations and map their processing time (ProcTime) as the key variable. A JSON mapping file would be generated to tell Factory Twin exactly which simulation objects correspond to which Insights Hub assets.
MindSphereInterface → Property Mapping → Select Objects → ProcTime → Export JSON
02
Create Digital Twin Model in Factory Twin
The JSON mapping file would be imported into the Factory Twin application in Insights Hub. For each FESTO station, a recipe would be written to calculate processing time using the "Calculate Process Time" pre defined recipe, which takes the time difference between the part arriving at the first proximity sensor and the end proximity sensor.
Factory Twin → Create Digital Twin Model → Upload JSON → Add Mapping → Calculate Process Time Recipe
03
Load Real Data into Plant Simulation
The Digital Twin model access token would be copied from Insights Hub and pasted into the MindSphere Interface in Plant Simulation. The real processing times from the physical FESTO run would be loaded into Plant Simulation through Model State: Restore. The simulation would then run with actual measured station times instead of planned estimates.
Plant Simulation → MindSphere Interface → Set Access Token → Load → Restore Model State
04
Upload Simulation Results Back to Insights Hub
After running the simulation with real data, the results would be uploaded back to Insights Hub. This would close the loop, meaning real data would improve the simulation, and the simulation results would be stored alongside the real data in the cloud for comparison, reporting, and continuous improvement.
Event Controller → Start Simulation → Give Result Name → Upload to Insights Hub
Key Terms — Industry 4.0 Glossary
Virtual AssetA digital representation of a physical machine or station in Insights Hub that stores variables mirroring real sensor data.
Physical AssetRepresents the connection method between physical hardware (PLC) and Insights Hub, in this case the MindConnect Nano gateway.
Aspect TypeA template containing variables, for example FM_start, FM_end, and FM_position for the Front Magazine station.
MindConnect NanoThe cloud gateway device that reads PLC datapoints from physical FESTO stations and streams them to Insights Hub at 1 second intervals.
Factory TwinAn Insights Hub application that maps physical assets to Plant Simulation objects, enabling real data to drive simulation updates automatically.
SimTalkThe programming language used inside Tecnomatix Plant Simulation. Here it was used with AI assistance to write Init methods and experiment parameter tables.
Digital TwinA live virtual model of the physical system that is updated with real data so the simulation reflects actual production performance, not just planned values (Grieves, 2014).
Closed LoopThe complete Industry 4.0 cycle: physical system to data to cloud to simulation to results and back to cloud. This project implements the full loop.
📋 Future Work — What Could Be Done Next
The current implementation successfully closes the IoT to simulation loop. However, several enhancements could be added in future iterations:
  • → Real time predictive maintenance using machine learning models trained on the historical telemetry data from Insights Hub (Susto et al., 2015).
  • → Automated simulation parameter updates where the Digital Twin writes improved processing times back to Plant Simulation without manual intervention.
  • → Integration of the Bambu P1S printers into the same IoT dashboard through their Bambu Cloud API, creating a unified view of the entire production system.
  • → Mobile alerting and remote control allowing engineers to stop faulty printers or adjust schedules directly from a smartphone.
