01 — The Big Picture
LOOP

The Industry 4.0
Closed Loop.

This is what separates digital manufacturing from traditional manufacturing. The physical FESTO CP Lab produces real data. That data flows into Siemens Insights Hub. Insights Hub feeds into Plant Simulation. The simulation results go back into Insights Hub. The loop is closed — the digital model improves continuously as the physical system runs.

01
Physical World
FESTO CP Lab
6 stations running physical assembly. Each station has a PLC (Programmable Logic Controller) generating real process data — start times, end times, positions.
Real hardware
02
Gateway
MindConnect Nano
The MindConnect Nano is the cloud gateway. It reads PLC datapoints from each station via IP address and streams the data to Siemens Insights Hub in real-time at a 1-second cycle.
1s data cycle
03
Cloud Platform
Insights Hub
Siemens Insights Hub stores and processes the live data. Virtual assets mirror each physical FESTO station. Factory Twin maps the data to the Plant Simulation model via calculated recipes.
Live cloud data
04
Simulation
Plant Simulation
Real processing times from Insights Hub are loaded into Tecnomatix Plant Simulation, updating the model with actual data. The simulation now reflects how the real system is performing, not just planned values.
Loop closed ✓
Why this matters for the Phones-R-Us contract
When 25 Bambu P1S printers are running 24/7, the IoT system doesn't just monitor — it enables safe unattended operation. Fire detection, security alarms, printer failure alerts, and performance monitoring all feed into the same connected ecosystem. If a printer fails at 3am, the system detects it, logs it, and alerts the engineer. The digital twin means the simulation is continuously updated with real production data, so capacity planning stays accurate as the system runs.
02 — Siemens Insights Hub
IH

Insights Hub
Setup.

Week 9 Workshop
Step 1 — Virtual Representation
Aspect Types & Asset Types
Created the virtual structure of each FESTO station in Insights Hub's Asset Manager. Each station gets an Aspect Type (contains variables), an Asset Type (template), and an Asset (the actual virtual station).
  • Created Aspect Type with 3 variables per station (start, end, position)
  • Created Asset Type from the aspect template
  • Created Asset — virtual twin of each physical station
  • Named using MMU convention: DM26_CPL_Station_X
Step 2 — Physical Connection
MindConnect Nano
Connected the physical FESTO PLC to Insights Hub via the MindConnect Nano gateway. Each station's IP address was mapped to a data source, with 3 datapoints per station streaming at 1-second intervals.
  • Configured data sources using PLC IP addresses per station
  • Station 1 (Front Magazine): 172.21.1.1
  • Station 2 (Measurement): 172.21.2.1
  • Stations 3–6 mapped at respective IP addresses
  • Data acquisition mode set to READ, 1s cycle time
Step 3 — Data Mapping
Joining Physical to Virtual
Mapped the physical datapoints (from MindConnect) to the virtual variables (in the Asset). This completes the connection — real PLC data now appears as virtual asset variables in the cloud.
  • Linked each physical datapoint to its corresponding virtual variable
  • All 6 stations mapped: FM, Measurement, iPick, Output, Press, Oven
  • Data now visible as live time-series in Insights Hub
  • Foundation for Factory Twin and Digital Twin built on top of this

03 — AI-Enhanced Simulation
AI

AI Running
Experiments.

Week 6 Workshop
AI-Enhanced Workflow
Using AI to generate SimTalk code and optimise the simulation
Rather than writing all the Tecnomatix SimTalk code manually, AI was used to accelerate the simulation build. The production system was described in detail to the AI — station processing times, worker shifts, parallel stations — and the AI generated the SimTalk code for setting up experiments and parameter tables. The outputs were then critically verified and adjusted before being applied, following good engineering practice.
01
Describe the system
Full production model described to AI — station processing times, worker pools, shifts, parallel stations, bottleneck analyser locations
02
AI generates SimTalk
AI produces SimTalk code for the Init method, parameter table structure, and experiment manager configuration — accelerating the build significantly
03
Verify & apply
AI output critically reviewed, tested, and corrected before applying. Not blindly accepted — engineering judgement applied to validate every output
What the Experiment Manager Was Used For
Experiment 1
How Many Workers?
The Experiment Manager was used to test different numbers of workers against throughput output. The simulation was run multiple times with the numWorkers variable set to different values, identifying the minimum worker count that still achieves the production target.
  • Variable: numWorkers tested across a range
  • Output: drain.StatNumIn — total parts completed
  • 5 observations per experiment for statistical validity
  • Result: optimal worker number identified from data
Experiment 2
Parameter Optimisation
A parameter table was built to test multiple variables simultaneously — processing times, setup times, parallel station configurations. AI generated the SimTalk Init method to read from the parameter table and apply values to each station.
  • Parameter table: processing times, setup times per station
  • AI-generated SimTalk reads table and applies to model
  • Multiple configurations tested in batch mode
  • Bottleneck analyser output compared across runs
Experiment 3
Bottleneck Analysis
Each station was linked to a bottleneck analyser in the simulation. Running experiments across different configurations revealed where throughput was most constrained — confirming Station 5 (Oven, 35s) as the FESTO line bottleneck.
  • Bottleneck analyser connected to every station
  • Station 5 Oven (35s) confirmed as primary constraint
  • Data confirmed total FESTO cycle within 150s takt
  • Simulation matched physical FESTO measurements ✓

04 — Digital Twin
DT

Factory Twin —
Closing the Loop.

Week 9 Workshop

The Digital Twin is the most advanced element of this project. Factory Twin is a Siemens Insights Hub application that creates a live two-way connection between the physical FESTO CP Lab and the Tecnomatix Plant Simulation model — the loop is fully closed.

01
Generate the Mapping File from Plant Simulation
The MindSphereInterface library was added to the Plant Simulation model. The interface was configured to select the FESTO stations and map their processing time (ProcTime) as the key variable. A JSON mapping file was generated — this tells Factory Twin exactly which simulation objects correspond to which Insights Hub assets.
MindSphereInterface → Property Mapping → Select Objects → ProcTime → Export JSON
02
Create Digital Twin Model in Factory Twin
The JSON mapping file was imported into the Factory Twin application in Insights Hub. For each FESTO station, a recipe was written to calculate processing time — using the "Calculate Process Time" pre-defined recipe which takes the time difference between the part arriving at the first proximity sensor and the end proximity sensor.
Factory Twin → Create Digital Twin Model → Upload JSON → Add Mapping → Calculate Process Time Recipe
03
Create a Study and Simulation Run
A Study was created within Factory Twin specifying which variables to monitor. A Simulation Run was then configured — specifying the time range of real FESTO data to use (e.g. 04/03/2024, 12:00–14:00) and running the simulation against that real data window.
Studies → Create Study → Simulation Runs → Set Time Range → Simulate
04
Load Real Data into Plant Simulation
The Digital Twin model access token was copied from Insights Hub and pasted into the MindSphere Interface in Plant Simulation. The real processing times from the physical FESTO run were loaded into Plant Simulation via Model State: Restore. The simulation now runs with actual measured station times, not planned estimates.
Plant Simulation → MindSphere Interface → Set Access Token → Load → Restore Model State
05
Upload Simulation Results Back to Insights Hub
After running the simulation with real data, the results were uploaded back to Insights Hub. This closes the loop — real data improved the simulation, and the simulation results are now stored alongside the real data in the cloud for comparison, reporting, and continuous improvement.
Event Controller → Start Simulation → Give Result Name → Upload to Insights Hub

Key Terms — Industry 4.0 Glossary
Virtual AssetDigital representation of a physical machine or station in Insights Hub. Stores variables that mirror real sensor data.
Physical AssetRepresents the connection method between physical hardware (PLC) and Insights Hub — in this case the MindConnect Nano gateway.
Aspect TypeA template containing variables — e.g. FM_start, FM_end, FM_position for the Front Magazine station.
MindConnect NanoThe cloud gateway device that reads PLC datapoints from physical FESTO stations and streams them to Insights Hub at 1-second intervals.
Factory TwinAn Insights Hub application that maps physical assets to Plant Simulation objects, enabling real data to drive simulation updates automatically.
SimTalkThe programming language used inside Tecnomatix Plant Simulation. Used here with AI assistance to write Init methods and experiment parameter tables.
Digital TwinA live virtual model of the physical system — updated with real data so the simulation reflects actual production performance, not just planned values.
Closed LoopPhysical system → Data → Cloud → Simulation → Results → back to Cloud. The complete Industry 4.0 cycle — implemented in this project.
